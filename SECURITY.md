# Безопасность

## Сообщить о проблеме

Не публикуйте потенциальную уязвимость в Issue. Напишите владельцу проекта по приватному каналу и приложите: краткое описание, шаги воспроизведения, затронутый URL или файл и возможные последствия. Не прикладывайте действующие пароли, токены или персональные данные.

## Модель защиты

- Supabase Auth отвечает за сессии и учетные записи.
- RLS ограничивает данные владельцем записи и правилами команды.
- Утвержденные планы и закрытые месяцы дополнительно защищены триггерами PostgreSQL.
- Приглашения команды имеют срок действия, одноразовый токен и при наличии email привязаны к этому адресу.
- Экспорт, импорт и приглашения ограничены rate limit на уровне Supabase.
- PWA не кеширует персональные HTML-страницы.

## Обязательная настройка Supabase

1. В `Authentication > URL Configuration` укажите production Site URL и разрешите только нужные Redirect URLs.
2. Включите подтверждение email и защиту от утечки паролей.
3. Установите минимальную длину пароля не ниже 12 символов.
4. Отключите anonymous sign-ins, если они не используются.
5. Для публичного запуска включите CAPTCHA для регистрации и восстановления пароля.
6. Никогда не добавляйте `service_role` в Netlify, клиентский код или GitHub Actions.
7. Применяйте все SQL-миграции по порядку до деплоя кода.

## Операционные правила

- Проверяйте GitHub Security alerts после каждого обновления зависимостей.
- При подозрении на утечку немедленно отзовите и замените ключи Supabase и активные сессии.
- Не используйте названия задач для хранения паролей, медицинских данных или иных секретов.
